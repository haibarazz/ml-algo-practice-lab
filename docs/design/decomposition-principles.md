# 模块拆解原则

这个项目的目标不是简单收集题目，而是把一个算法或开源项目拆成可以反复训练的手撕模块。

## 为什么这样拆

面试和真实项目学习通常卡在三个层面：

- 原理知道，但不能从零写出代码。
- 能看懂答案，但无提示时写不出来。
- 能读单个函数，但不知道它在系统里承担什么角色。

因此每个主题要同时覆盖：

- 数学定义：公式、变量含义、边界条件。
- 手写实现：不依赖大型框架的最小核心实现。
- 测试验证：用少量具体输入输出锁定核心正确性。
- 面试追问：复杂度、数值稳定性、工程取舍。
- 系统位置：它在完整 ML/LLM 项目中的作用。

## 模块层级

推荐把内容拆成四层：

1. `primitive`: 基础算子、距离、概率、损失、矩阵计算。
2. `algorithm`: KNN、KMeans、回归、树模型、优化器、注意力等完整算法。
3. `training`: SFT、DPO、LoRA、数据整理、评估等训练流程。
4. `system`: 数据管线、训练入口、推理服务、模型发布、监控评估。

## 开源项目拆解规则

开源项目拆解优先按学习路径组织，而不是照搬原仓库目录。每个模块必须保留 `Source Mapping`，标出它对应的真实源码文件、配置文件或脚本。

最小拆解粒度是可手写练习，而不是源码阅读笔记。除系统架构模块外，每个拆解模块都应该能落到一个函数、类、损失、数据处理步骤或推理组件的 TODO 实现。

不同开源项目必须隔离在 `projects/` 下的独立目录中，例如 `projects/minimind/`、`projects/nanogpt/`。通用机器学习和深度学习手撕算法不放进 `projects/`，统一放在 `ml_dl_handwriting/`。

推荐顺序：

1. 项目总览和执行路径
2. Tokenizer 与数据格式
3. Dataset 与 Collator
4. 模型结构
5. Loss 与训练目标
6. 预训练循环
7. SFT
8. DPO / 偏好对齐
9. 推理与采样
10. 评估
11. 系统架构总图

## 每个模块必须回答的问题

- 这个模块解决什么问题？
- 输入和输出是什么？
- 数学上最小必要定义是什么？
- 带提示版本应该提供哪些台阶？
- 无提示版本应该隐藏哪些提示？
- 测试应该覆盖哪些边界情况？
- 面试官可能追问什么？
- 它在真实项目中的位置是什么？

## 测试原则

测试保持轻量，优先学习反馈而不是工程覆盖率：

- 每个模块一个 `tests.py`。
- 直接使用 `assert` 和固定样例。
- 能暴露核心公式、shape、边界条件或常见实现错误即可。
- 不默认引入 pytest、CI、mock、fixture 等复杂机制。
- 后续如果某个模块变成真实库代码，再单独升级测试方式。

## 练习形态

本项目不采用 ACM / OJ 题型。默认练习形态参考 `llm-algo-leetcode`：

- 给出背景和核心公式。
- 先给带提示练习区，提供函数或类的代码框架和 TODO 台阶。
- 再给无提示练习区，只保留目标、签名和测试要求。
- 提供测试函数验证实现。
- 使用 `STOP HERE` 分隔题目区和答案区。
- 答案区包含参考代码、逐步解析和工程要点。

## 文件形态

每个模块同时维护 Notebook 和 Markdown：

- Notebook 用于交互式练习、运行 TODO、执行测试。
- Markdown 用于快速阅读、版本 review 和未来发布。
- 两者内容应保持一致，优先从 Notebook 生成或同步 Markdown，避免长期手工分叉。
- 带提示区、无提示区和答案区保留在同一个模块页面内，使用明确标题和 `STOP HERE` 分隔。

标准页面顺序：

1. 原理最小说明
2. 带提示练习区
3. 无提示练习区
4. 测试区
5. `STOP HERE`
6. 参考答案与解析
